Feature Request Triage Cheat Sheet

ใช้ทุกครั้งที่ลูกค้าขอ feature. เป้าหมายคือรับฟังจริง แต่ไม่ให้ roadmap กลายเป็นรายการงาน custom ของลูกค้าคนเดียว.

แก่นคิด

Feature request ไม่ใช่คำสั่งให้ build. มันคือ clue ว่าลูกค้าติด job/pain อะไร.

ถามหา past behavior, workaround, impact, frequency, และ willingness-to-pay ก่อนเชื่อว่า request นี้ควรเข้า roadmap.

4 decisions

Decisionใช้เมื่อคำตอบ
Build now ICP หลายรายขอ, กระทบ retention/revenue, manual proof แล้ว, scope เล็ก ทำ scope แรกที่เล็กที่สุดและวัด repeat usage หลังปล่อย
Manual first pain น่าสนใจ แต่ pattern/solution ยังไม่ชัด ทำ concierge/workaround 1-3 รอบก่อน automate
Later with gate น่าจะมีประโยชน์ แต่หลักฐานยังไม่พอหรือ scope ยังใหญ่ ตั้ง evidence gate เช่น ICP 3 รายขอซ้ำ หรือ manual proof 2 รอบ
Say no หลุด ICP, เพิ่ม complexity สูง, model/channel แย่ลง, ไม่มี willingness-to-pay ปฏิเสธอย่างสุภาพและเสนอ workaround ถ้ามี

Questions before build

  1. ตอนนี้คุณแก้ปัญหานี้ยังไง?
  2. ปัญหานี้เกิดบ่อยแค่ไหน และ cost คืออะไร?
  3. ถ้าไม่มีสิ่งนี้ คุณยังใช้ product ต่อไหม?
  4. ใครอีกใน segment เดียวกันเจอปัญหานี้?
  5. คุณยอมใช้ manual workaround รอบนี้ไหม?
  6. ถ้าทำแล้วจะช่วยให้ต่ออายุ/เพิ่ม usage/จ่ายเพิ่มยังไง?

Scoring shortcut

คะแนนบวก

  • ตรง ICP หลัก
  • มีลูกค้า 3+ รายเจอ pain เดียวกัน
  • ปลดล็อก retention/revenue
  • manual proof แล้ว
  • scope เล็ก
  • ช่วย pricing/channel/onboarding

คะแนนลบ

  • ลูกค้าคนเดียวขอ
  • หลุด ICP
  • ใหญ่เกิน 1-2 สัปดาห์
  • เพิ่ม AI cost แต่ราคาไม่เพิ่ม
  • ทำให้ onboarding ซับซ้อน
  • ดีต่อ demo แต่ไม่ช่วยใช้ซ้ำ

Reply templates

  • Build: "เราจะทำ scope แรกแบบเล็กที่สุดก่อน เพื่อให้ workflow นี้เกิดจริงและวัดผลได้."
  • Manual: "ก่อน build ผมขอช่วยทำ workaround รอบนี้ก่อน เพื่อดูว่า outcome คุ้มและ pattern ซ้ำไหม."
  • Later: "เราจะกลับมาดูเมื่อมีลูกค้า segment เดียวกันขอซ้ำหรือมี manual proof ชัด."
  • No: "ตอนนี้เรายังไม่รับทำส่วนนี้ เพราะจะพา product ออกจาก use case หลักที่กำลังโฟกัส."
