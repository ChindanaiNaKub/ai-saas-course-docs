Productized Wedge Cheat Sheet

ใช้ตอนคุณมี pilot/manual work แล้วเริ่มเห็น pattern. เป้าหมายคือเลือก workflow หนึ่งอันที่ขายซ้ำได้ และกัน product ไม่ให้กลายเป็น custom service.

แก่นคิด

Productize ไม่ได้แปลว่า automate ทุกอย่าง. Productize แปลว่าเลือก workflow ที่ซ้ำได้ แล้วทำให้มันเชื่อถือได้.

6 signals

Signalถามตัวเองผ่านเมื่อ
Same buyer คนซื้อ/คนใช้เป็น persona เดียวกันไหม พูด message เดียวแล้วเข้าใจ
Same trigger ปัญหาเกิดซ้ำจากเหตุการณ์เดียวกันไหม มี weekly/monthly/daily repeat event
Same input ข้อมูลที่ต้องใช้คล้ายกันไหม ทำ checklist/template ได้
Same output ผลลัพธ์ที่ต้องส่งมี structure ซ้ำไหม มี report/action list/workflow เดียวกัน
Same action ลูกค้าเอา output ไปทำ action คล้ายกันไหม วัด outcome ได้ด้วย metric คล้ายกัน
Same price logic คิดราคาด้วย value driver เดียวกันได้ไหม เช่น reports, volume, seats, recovered revenue

Boundary statement

เราทำสำหรับ [ICP] เพื่อช่วย [job] โดยรับ [input] แล้วส่ง [output]. ใน v1 เรายังไม่ทำ [not doing].

Good boundary

ชี้ ICP, repeat trigger, input, output, action และสิ่งที่ไม่ทำ.

Bad boundary

กว้างแบบ platform, รับทุก data source, ทำทุก dashboard, ช่วยทุกธุรกิจ.

Roadmap rule

Build เฉพาะสิ่งที่ทำให้ wedge นี้:

  • เร็วขึ้น: time-to-value ลดลง
  • ซ้ำได้ขึ้น: repeat event ง่ายขึ้น
  • เชื่อถือได้ขึ้น: error/support ลดลง
  • ขายง่ายขึ้น: proof/message/pricing ชัดขึ้น
  • ต่ออายุง่ายขึ้น: outcome เกิดซ้ำและ stakeholder เห็น value

Do not productize yet if

  • ลูกค้าแต่ละรายเป็นคนละ ICP
  • input/output ต้อง custom ใหม่ทุกครั้ง
  • ไม่มี repeat trigger
  • ลูกค้าอ่าน output แต่ไม่ action
  • pricing ต้อง negotiate ใหม่ทุกเคส
