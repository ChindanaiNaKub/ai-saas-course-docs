Lesson 0006 · Evidence board

ตัดสินใจจาก evidence: build, iterate, pivot, หรือ kill

บทเรียนนี้สอนทักษะเดียว: รวมสัญญาณจาก interviews, pilots, money, และ distribution เพื่อเลือก next move โดยไม่กลับไป build จากความหวังล้วนๆ.

กับดัก

ได้ reply นิดหน่อยแล้วดีใจจน build หรือโดน reject แล้วคิดว่า idea แย่ทันที ทั้งสองอย่างคืออารมณ์นำ evidence.

เป้าหมาย

แยก signal ตาม 4 ด้าน: pain, outcome, money, distribution แล้วตัดสินใจว่าจะ automate อะไร หรือควรปรับอะไร.

1. ทำไมต้องมี decision gate?

หลังผ่านบทเรียน 1-5 คุณมีเครื่องมือครบ: distribution-first idea, customer interview, manual pilot, paid offer, และ distribution sprint. แต่ founder ยังพลาดได้ที่จุดนี้ เพราะอยาก build มากจนมองข้าม signal อ่อน หรือกลัว rejection จน kill เร็วเกินไป.

Decision gate คือ checkpoint ที่บังคับให้ถามว่า "เราเรียนรู้อะไรจากของจริง?" ก่อนตัดสินใจ. แนวคิดนี้เข้ากับ Lean Startup เรื่อง validated learning และคำแนะนำของ YC เรื่อง launch, talk to users, build what people want (YC Essential Startup Advice).

อย่า build เพื่อหนีความไม่แน่นอน. Build เฉพาะเมื่อ evidence บอกว่าการ automate ส่วนหนึ่งจะเพิ่มการเรียนรู้หรือยอดขายเร็วขึ้น.

2. Evidence 4 ด้าน

ด้านคำถามสัญญาณที่อยากเห็น
Customer pain ปัญหานี้เกิดจริง บ่อย และเจ็บไหม? recent events, cost, workaround, urgency
Outcome/pilot ผลลัพธ์ที่เราส่งมอบถูกใช้ใน workflow จริงไหม? ส่งข้อมูล, ใช้ report, ขอแก้/ใช้ต่อ, นัด review
Money มี willingness to pay หรือ budget path ไหม? จ่าย, มัดจำ, invoice, budget owner, serious negotiation
Distribution เราหาลูกค้ากลุ่มนี้ซ้ำได้ไหม? lead list, replies, calls, pilots จาก channel เดิม

3. Decision matrix

Build small automation

เมื่อ pain, outcome, money, distribution เริ่มซ้ำ และ manual workflow มีส่วนที่ทำซ้ำจนเสียเวลา.

Iterate

เมื่อ pain มีจริง แต่ offer, message, price, หรือ pilot output ยังไม่ convert.

Pivot

เมื่อ evidence ชี้ว่าลูกค้าจริง, use case จริง, หรือ channel จริง ไม่ตรงกับสมมติฐานเดิม.

Kill

เมื่อหลายรอบแล้วยังไม่มี pain ล่าสุด, ไม่มี commitment, ไม่มีคนจ่าย, หรือ cost to serve แพงกว่า value.

4. ถ้าจะ build ให้ build แค่ friction ที่พิสูจน์แล้ว

สมมติ manual pilot คือ AI lost order report จากแชท LINE. ถ้าลูกค้าจ่ายและส่งแชทจริง สิ่งที่ควร build แรกอาจไม่ใช่ dashboard แต่เป็น script/importer ที่ช่วยคุณทำ report เร็วขึ้น.

Manual pain ของคุณAutomation แรกที่ดียังไม่ควร build
copy chat เข้า AI เสียเวลามาก simple parser/import template full dashboard + roles + billing
report format ต้องทำซ้ำทุก pilot report generator + reusable template custom theme editor
ลูกค้าไม่เข้าใจ insight ถ้าไม่มี review call guided review checklist self-serve onboarding เต็มระบบ

5. แบบฝึกหัด: Evidence board

ใส่ evidence ที่มีตอนนี้ แล้วกดประเมิน. ตัวประเมินนี้เป็น JavaScript ในไฟล์ ไม่เรียก AI และไม่ส่งข้อมูลออกไปไหน.

รอ evidence ของคุณอยู่.

6. Build-scope guardrail

ถ้าผลคือควร build ให้ใช้ตัวนี้ตัด scope. เขียนสิ่งที่อยาก build แล้วดูว่ามันใกล้กับ evidence หรือเป็นความอยากล้วนๆ.

รอ scope ของคุณอยู่.

7. Mini Quiz

สถานการณ์: คุณคุย 10 คน มี 6 คนเจอ pain จริง, 2 คนส่งข้อมูลให้ทำ pilot, 1 คนจ่าย 1,500 บาท, แต่ outreach 50 รายได้ reply แค่ 2. ควรทำอะไร?

เลือกคำตอบหนึ่งข้อ.

8. Homework ก่อนบทเรียน 7

เอา idea/pilot ของคุณมาเขียน evidence board 4 ด้าน:

  1. Pain: หลักฐานจากลูกค้า
  2. Outcome: ผลจาก manual pilot
  3. Money: เงินหรือ budget signal
  4. Distribution: lead/reply/call/pilot จาก channel

ส่งกลับมาให้ผมดูได้เลย ผมจะช่วยอ่านว่า next move ควร build, iterate, pivot, หรือ kill และถ้า build ควร build แค่ไหน.
