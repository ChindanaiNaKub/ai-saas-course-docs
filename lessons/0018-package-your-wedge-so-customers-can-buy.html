Lesson 0018 · Packaging the wedge

ทำให้ลูกค้าซื้อ wedge ได้ง่าย

บทเรียนนี้สอนทักษะเดียว: เปลี่ยน wedge ที่ชัดแล้วให้เป็น package ที่ขายซ้ำได้ มีราคา, ขอบเขต, usage limit, onboarding, และ expansion path ที่ไม่ต้องคิดใหม่ทุกดีล.

กับดัก

คุณรู้แล้วว่า product ช่วยใครและช่วยเรื่องอะไร แต่ทุกครั้งที่ขายยังต้อง quote ใหม่ scope ใหม่ promise ใหม่ จนลูกค้าซื้อยากและคุณส่งมอบยาก.

เป้าหมาย

มี package 1-2 แบบที่ลูกค้าเข้าใจใน 60 วินาที: ได้อะไร, ราคาเท่าไร, เริ่มยังไง, ใช้ได้แค่ไหน, ถ้าโตแล้วเพิ่มอะไร.

1. Pricing ไม่ใช่แค่ตัวเลข

บทเรียนที่ 4 พูดเรื่อง willingness-to-pay และบทเรียนที่ 11 พูดเรื่อง unit economics. ตอนนี้เราต้องรวมสองอย่างนี้เข้ากับ wedge: package ต้องสื่อ value ให้ลูกค้าซื้อได้ง่าย และป้องกันไม่ให้ต้นทุน AI/support บาน. Stripe พูดถึง SaaS metrics เช่น churn, retention, expansion, unit economics เพราะ package ที่ดีส่งผลต่อทุกตัวนี้.

Package ที่ดีทำให้ลูกค้ารู้ว่าเขาซื้อ outcome อะไร ไม่ใช่ซื้อ feature กองหนึ่ง.

2. Package มี 5 ชิ้น

ชิ้นส่วนต้องตอบอะไรตัวอย่าง
Outcome ลูกค้าจะได้ progress อะไรซ้ำๆ weekly lost-order report พร้อม action list
Scope รวมอะไรและไม่รวมอะไร รวม 2 channels, ไม่รวม full CRM/integration custom
Usage limit ใช้ได้กี่ reports, records, seats, workflows สูงสุด 4 reports/month และ 5,000 chat messages
Onboarding เริ่มใช้ยังไง ใช้เวลานานแค่ไหน ใครต้องทำอะไร setup call 45 นาที + first report ภายใน 48 ชม.
Expansion path ถ้า value เพิ่ม ลูกค้าจ่ายเพิ่มเพราะอะไร เพิ่ม channels, volume, teams, advanced review

3. เลือก price metric ให้ตรง value driver

AI-era SaaS มักไม่เหมาะกับ seat-based อย่างเดียว เพราะต้นทุนและ value อาจผูกกับ usage หรือ outcome. Bain และ a16z พูดถึงการขยับไป hybrid, usage-based, หรือ outcome-based pricing มากขึ้น. สำหรับ early founder อย่าซับซ้อนเกินไป ให้เลือก metric ที่ลูกค้าเข้าใจและต้นทุนคุณควบคุมได้.

Per workflow/report

ดีเมื่อ value เกิดเป็นรอบชัด เช่น weekly report, monthly audit, campaign review.

Usage tier

ดีเมื่อ cost/value โตตาม volume เช่น messages, documents, orders, tickets.

Hybrid

base subscription + usage cap/overage. เหมาะกับ AI product ที่มีต้นทุนแปรผัน.

4. แพ็กเกจแรกควรมีน้อย

ช่วงแรกอย่าทำ pricing page ที่มี 5 tiers เพราะคุณยังเรียนรู้. ใช้ 1 paid pilot package + 1 recurring package ก็พอ. จุดประสงค์คือให้ลูกค้าตัดสินใจง่ายและให้คุณเรียนรู้ว่า scope/price/usage limit ไหนขายได้.

Simple ladder

  1. Paid setup/pilot: prove first value ภายใน 1-2 สัปดาห์
  2. Monthly core: repeat workflow รายเดือนพร้อม usage cap
  3. Expansion: เพิ่ม volume/team/channel เมื่อ value ชัด

Too much too early

  • Free plan ที่ดึงแต่ non-buyers
  • custom quote ทุกเคส
  • unlimited AI usage
  • enterprise feature ก่อนมี repeatable core

5. แบบฝึกหัด: Package sanity check

กรอก package candidate แล้วระบบจะเช็กว่าซื้อง่ายและส่งมอบได้ไหม. ตัวประเมินนี้เป็น JavaScript ในไฟล์ ไม่เรียก AI และไม่ส่งข้อมูลออกไปไหน.

รอข้อมูล package อยู่.

6. สร้าง package draft

กรอกข้อมูลสั้นๆ แล้วระบบจะร่าง package ที่คุณเอาไปใช้ใน sales call ได้.

กรอก package แล้วกดร่าง.

7. Mini Quiz

สถานการณ์: AI product ของคุณมี cost ตามจำนวนข้อความที่วิเคราะห์ แต่คุณอยากทำ package "unlimited analysis" เพื่อให้ลูกค้ารู้สึกคุ้ม. อะไรคือความเสี่ยงหลัก?

เลือกคำตอบหนึ่งข้อ.
