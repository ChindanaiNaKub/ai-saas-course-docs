Lesson 0016 · Feature request triage

รับฟังลูกค้าโดยไม่กลายเป็นทีม custom dev

บทเรียนนี้สอนทักษะเดียว: แยก feature request ว่าอะไรควร build, อะไรควรทำ manual ก่อน, อะไรควรตอบ no, และอะไรคือสัญญาณว่า product กำลังหลุด ICP.

กับดัก

ลูกค้าคนแรกๆ ขอ feature แล้วคุณรีบ build เพราะกลัวเสียลูกค้า สุดท้าย product หนักขึ้น แต่ positioning, onboarding, pricing, และ distribution แย่ลง.

เป้าหมาย

ใช้ request เป็นหลักฐานการเรียนรู้ ไม่ใช่คำสั่ง. คุณจะตอบได้ว่า "ตอนนี้ทำ", "ทดสอบ manual ก่อน", "เก็บไว้", หรือ "ไม่ทำ".

1. ลูกค้าพูด solution แต่คุณต้องหา job

ลูกค้ามักพูดว่าอยากได้ feature เช่น export Excel, dashboard, integration, alert, role permission. แต่สิ่งที่คุณต้องรู้คือเขาพยายามทำงานอะไรให้คืบหน้า. Jobs to Be Done ช่วยให้เรามองผ่านคำขอไปเห็น progress ที่ลูกค้าต้องการ. The Mom Test ก็เตือนเหมือนกัน: อย่ารับคำชม/ไอเดียเป็นหลักฐาน ให้ถามพฤติกรรมและปัญหาจริง.

Feature request ไม่ใช่ roadmap. Feature request คือ clue ว่าลูกค้าติดอะไร.

2. คำถามก่อนรับ request

ถามเพื่อดูอะไรคำตอบที่มีน้ำหนัก
"ตอนนี้คุณแก้ปัญหานี้ยังไง?" มี pain จริงหรือแค่อยากได้ มี workaround ที่เสียเวลา/เสียเงิน/พลาดบ่อย
"ถ้าไม่มี feature นี้ คุณยังใช้ product ต่อไหม?" กระทบ retention หรือแค่ nice-to-have กระทบการใช้ซ้ำ/renewal ชัด
"ใครอีกที่เจอปัญหานี้?" เป็น pattern ใน ICP หรือ one-off ลูกค้าหลายรายใน segment เดียวกันเจอเหมือนกัน
"ถ้าเราช่วยแบบ manual รอบนี้ก่อน โอเคไหม?" ทดสอบ value ก่อน build ลูกค้ายอมใช้ manual workaround เพื่อให้ outcome เกิด

3. Build / Manual / Later / No

Build now

request มาจาก ICP หลายราย, กระทบ retention/revenue, manual พิสูจน์แล้ว, scope เล็กพอ.

Manual first

pain น่าสนใจแต่ยังไม่รู้ pattern. ทำด้วยมือ 1-3 รอบเพื่อดูว่าควร automate ตรงไหน.

Say no

หลุด ICP, ทำให้ product ซับซ้อน, ไม่มี willingness-to-pay, หรือแก้ปัญหาลูกค้าคนเดียว.

คำว่า "later" ใช้ได้ แต่ต้องมีเงื่อนไข: จะกลับมาดูเมื่อมีลูกค้า ICP กี่รายขอซ้ำ, มี revenue impact เท่าไร, หรือมี manual proof แบบไหน.

4. Feature ที่ดีต้องช่วย distribution หรือ retention

AI ทำให้ build ง่ายขึ้น แต่นั่นทำให้การเลือก build ยิ่งสำคัญ. Brian Balfour พูดเรื่อง product-channel fit และ four fits: product, market, channel, model ต้องไปด้วยกัน. Feature ที่ดีสำหรับ early SaaS ไม่ใช่แค่ "ทำได้" แต่ต้องทำให้ลูกค้าใช้ซ้ำง่ายขึ้น, ขายง่ายขึ้น, onboarding สั้นลง, หรือ pricing แข็งแรงขึ้น.

Feature ที่น่าทำ

  • ลด time-to-value
  • เพิ่ม repeat event
  • ลด support burden ที่เกิดซ้ำ
  • ปลดล็อก paid pilot/renewal
  • ทำให้ case study/referral ชัดขึ้น

Feature ที่น่าระวัง

  • มีลูกค้าคนเดียวขอ
  • ทำให้ onboarding ยาวขึ้น
  • เปลี่ยน ICP แบบไม่ตั้งใจ
  • เพิ่ม AI cost แต่ราคาไม่เพิ่ม
  • ดีต่อ demo แต่ไม่เพิ่มการใช้จริง

5. แบบฝึกหัด: Feature triage score

ให้คะแนน request หนึ่งอัน. ตัวประเมินนี้เป็น JavaScript ในไฟล์ ไม่เรียก AI และไม่ส่งข้อมูลออกไปไหน.

รอ feature request อยู่.

6. ร่างคำตอบลูกค้า

เลือก decision แล้วระบบจะร่างข้อความตอบลูกค้าแบบรักษา trust.

เลือก decision แล้วกดร่างคำตอบ.

7. Mini Quiz

สถานการณ์: ลูกค้าใหญ่หนึ่งรายขอ integration เฉพาะระบบภายใน ใช้เวลา 3 สัปดาห์ และลูกค้า ICP รายอื่นยังไม่เคยขอ แต่เขาบอกว่า "ถ้ามีอาจจะต่อ". คุณควรทำอะไร?

เลือกคำตอบหนึ่งข้อ.
