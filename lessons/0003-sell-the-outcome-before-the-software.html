Lesson 0003 · MVP without overbuilding

ขายผลลัพธ์ก่อนสร้างซอฟต์แวร์

บทเรียนนี้สอนทักษะเดียว: ออกแบบ manual pilot 7 วัน เพื่อพิสูจน์ว่าลูกค้าต้องการผลลัพธ์จริง ก่อนคุณเสียเวลา build SaaS เต็มตัว.

กับดักเดิม

คุยแล้วลูกค้าดูสนใจ → เรากลับไป build dashboard, auth, billing, settings, integration → ผ่านไป 2 เดือน ยังไม่รู้ว่าเขาจะใช้จริงไหม.

ทางที่ดีกว่า

รับปาก outcome เล็กๆ → ส่งมอบแบบ manual → วัดว่าเขาใช้, จ่าย, ส่งข้อมูล, และขอนัดต่อไหม → ค่อย automate ส่วนที่ซ้ำ.

1. MVP ไม่ใช่ product เวอร์ชันเล็กเสมอไป

หลายคนเข้าใจว่า MVP คือ "ทำ product ให้ feature น้อยลง". บางครั้งถูก แต่สำหรับ founder ที่เคยเสียเวลา build 2-3 เดือน จุดที่ดีกว่าคือทำ manual pilot: ส่งมอบผลลัพธ์ด้วยมือก่อน แล้วค่อยดูว่าควรสร้างอะไร.

Lean Startup Co. อธิบาย MVP ว่าเป็น version ที่ช่วยให้ทีมเก็บ validated learning ได้มากที่สุดด้วย effort น้อยที่สุด. แปลเป็นภาษาคนสร้าง SaaS คือ: ถ้าคุณเรียนรู้ได้โดยใช้ Google Sheet, AI chat, manual report, และ call 20 นาที ก็ยังไม่ต้อง build ระบบเต็ม.

MVP ที่ดีไม่ได้ถามว่า "เราสร้างน้อยสุดได้แค่ไหน" แต่ถามว่า "เราต้องเรียนรู้อะไรที่เสี่ยงที่สุด และพิสูจน์ด้วยแรงน้อยสุดได้ยังไง".

2. Manual pilot คืออะไร?

Manual pilot คือการทำหน้าที่แทน software แบบจำกัดขอบเขตให้ลูกค้าจริง 1-3 ราย โดยลูกค้ารู้ว่ามันยัง manual หรือกึ่ง manual. เป้าหมายไม่ใช่แกล้งทำว่ามีระบบแล้ว เป้าหมายคือพิสูจน์ว่า outcome มี value จริง.

อยากทำ SaaSManual pilot ที่ควรทำก่อนยังไม่ต้อง build
AI สรุปแชท LINE ร้านค้า รับแชท 100 รายการ → สรุปคำถามซ้ำ, lost order, response issue → ส่ง report ภายใน 24 ชม. LINE OA integration, dashboard, user roles
AI report สำหรับ agency รับ campaign data 1 ลูกค้า → ทำ executive summary + insight 5 ข้อ → ส่งให้ account manager ใช้ประชุม template builder, API sync, multi-client portal
ระบบติดตาม no-show คลินิก รับ appointment sheet 1 สัปดาห์ → ส่ง reminder list + no-show pattern + call script booking system, patient app, staff permissions

YC มีบทเรียนเรื่อง How to Plan an MVP และ How to Build an MVP ที่ย้ำเรื่อง scope แคบและเรียนรู้จาก users เร็วที่สุด.

3. Manual pilot 7 ช่อง

ก่อน build อะไร ให้เขียนคำตอบ 7 ช่องนี้:

  1. Customer: ลูกค้ารายแรกคือใคร ชื่อ/role/ธุรกิจชัดไหม
  2. Promise: ผลลัพธ์ที่สัญญาจะส่งใน 7 วันคืออะไร
  3. Input: ต้องขอข้อมูลอะไรจากลูกค้า
  4. Manual workflow: เราจะทำเองด้วย tool ง่ายๆ ยังไง
  5. Delivery: ส่งงานเป็นอะไร report, file, call, dashboard mock, LINE message
  6. Commitment: ลูกค้าต้องจ่าย/มัดจำ/ส่งข้อมูล/นัดต่อ/intro อะไร
  7. Decision rule: เห็น signal แบบไหนถึง build, continue, หรือ kill

4. ขอบเขตที่ดี: เล็กแต่มีฟัน

ดี

  • ลูกค้า 1-3 ราย
  • ระยะเวลา 3-7 วัน
  • output ใช้ใน workflow จริง
  • มี commitment ก่อนเริ่ม
  • มี metric ชัด เช่น save 3 hours, recover 10 leads, reduce no-show

เสี่ยงกลับไป build-first

  • เริ่มจาก "ทำระบบก่อนแล้วค่อยหาลูกค้า"
  • ต้องใช้ integration หลายตัวก่อนเรียนรู้ได้
  • output เป็น demo ที่ไม่มีใครใช้ทำงานจริง
  • ไม่มีใครยอมส่งข้อมูลหรือจ่ายอะไรเลย
  • ทุกลูกค้าต้อง custom คนละแบบจนไม่เห็น product pattern

5. แบบฝึกหัด: ออกแบบ manual pilot

กรอก 5 ช่องนี้ แล้วกดประเมิน. ระบบนี้ยังเป็น JavaScript rule-based ในไฟล์ ไม่ได้เรียก AI หรือส่งข้อมูลออกไปไหน.

รอคำตอบของคุณอยู่.

6. Build Threshold Calculator

ให้คะแนน signal จาก pilot แบบเร็วๆ. ถ้ายังไม่ถึง threshold อย่าเพิ่ง build product เต็ม.

ใส่ตัวเลขหลังทำ pilot.

7. Mini Quiz

สถานการณ์: ลูกค้า 1 รายบอกว่าสนใจ AI สรุปแชท LINE แต่ยังไม่เคยส่งข้อมูลจริงหรือจ่ายเงิน คุณควรทำอะไร?

เลือกคำตอบหนึ่งข้อ.

8. Homework ก่อนบทเรียน 4

เลือก 1 idea แล้วเขียน manual pilot 7 ช่องจากบทนี้. ถ้าคุณยังไม่มีลูกค้าจริง ให้เลือก segment ที่จะหา 5 คนภายใน 48 ชั่วโมงก่อน.

เอา draft มาให้ผมดูได้เลย ผมจะช่วยตัด scope ให้เล็กลงจนทำได้จริงใน 7 วัน และบอกว่าอะไรควร manual, อะไรยังไม่ควร build.
