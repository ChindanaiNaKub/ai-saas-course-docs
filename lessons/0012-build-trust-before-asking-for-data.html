Lesson 0012 · Trust and data handling

สร้าง trust ก่อนขอข้อมูลลูกค้า

บทเรียนนี้สอนทักษะเดียว: ออกแบบ data handling สำหรับ AI pilot/SaaS ให้ลูกค้ากล้าให้ข้อมูล โดยไม่เก็บเกิน ใช้เกิน หรือเสี่ยงเกินจำเป็น.

กับดัก

ขอ export แชท/ไฟล์ลูกค้าทั้งหมดโดยไม่บอกว่าจะใช้ยังไง เก็บที่ไหน ส่งให้ AI provider ไหน และลบเมื่อไหร่.

เป้าหมาย

ขอข้อมูลน้อยที่สุด อธิบาย purpose, redaction, storage, retention, provider, และ human review ได้ชัดใน 60 วินาที.

1. Trust คือ distribution advantage

ถ้าคุณขาย AI SaaS ที่แตะข้อมูลจริง เช่น แชท LINE, ข้อมูลลูกค้า, appointment, campaign data, invoice, หรือไฟล์ภายในบริษัท ลูกค้าจะถามเรื่อง trust ไม่ช้าก็เร็ว. โดยเฉพาะตลาดไทยที่ relationship และความไว้ใจสำคัญมาก การตอบเรื่องข้อมูลได้ชัดอาจเป็นข้อได้เปรียบเหนือ "AI tool" ทั่วไป.

บทนี้ไม่ใช่คำปรึกษากฎหมายแทนทนาย แต่เป็น founder checklist. ถ้าคุณทำงานกับข้อมูลส่วนบุคคลจำนวนมาก, ข้อมูลอ่อนไหว, health/financial data, หรือขาย enterprise จริงจัง ควรปรึกษาผู้เชี่ยวชาญด้านกฎหมาย/ความปลอดภัย.

ลูกค้าไม่ได้กลัว AI อย่างเดียว เขากลัวคุณไม่รู้ว่าข้อมูลของเขาจะไปไหนและถูกใช้ยังไง.

2. หลัก 5 ข้อก่อนขอข้อมูล

  1. Purpose limitation: ขอเพื่อทำ outcome ไหนเท่านั้น
  2. Data minimization: ขอเฉพาะข้อมูลที่จำเป็นจริง
  3. Redaction first: ให้ลูกค้าลบ/ปิดข้อมูลส่วนตัวหรือ sensitive ก่อนส่งได้
  4. Retention window: บอกว่าจะลบเมื่อไหร่
  5. Provider transparency: บอกว่าใช้ AI/provider อะไร และข้อมูลถูกส่งออกไปไหม

หลัก data minimization จาก EDPS คือเก็บเฉพาะข้อมูลที่เกี่ยวข้องและจำเป็นกับ purpose นั้น และเก็บเท่าที่จำเป็น. GDPR Article 5 ก็พูดถึง principles เช่น purpose limitation, data minimization, storage limitation, integrity และ confidentiality. สำหรับไทย PDPA มีแนวคิดเรื่องหน้าที่ของ controller/processor และสิทธิของเจ้าของข้อมูลตาม overview ของ DLA Piper.

3. AI-specific trust risks

Riskตัวอย่างใน SaaSGuardrail แรก
Prompt injection แชทลูกค้ามีคำสั่งแฝงให้ model เปิดเผย prompt หรือข้าม policy จำกัดสิทธิ์ model, ไม่ให้ model มี tool access เกินจำเป็น, validate output
Sensitive information disclosure report เผลอแสดงเบอร์โทร/email/ที่อยู่ลูกค้าปลายทาง redact input, filter output, role-based access
Insecure output handling เอา output จาก AI ไปสร้าง SQL/HTML/action โดยไม่ validate sanitize, review, allowlist actions
Overreliance ลูกค้าใช้ AI recommendation ตัดสินใจเรื่องสำคัญโดยไม่ตรวจ human review, confidence label, explain source evidence

OWASP Top 10 for LLM Applications ระบุ risk เช่น prompt injection, insecure output handling, และ sensitive information disclosure ซึ่งสำคัญมากเมื่อ product แตะข้อมูลจริง.

4. คำตอบเรื่อง "เอาข้อมูลไป train ไหม?"

คุณต้องตอบตาม provider และ contract ที่คุณใช้จริง. ตัวอย่างเช่น OpenAI ระบุว่าโดย default business products รวมถึง API Platform ไม่ได้นำ inputs/outputs ไป train models เว้นแต่ลูกค้า opt in แชร์ข้อมูล. แต่ถ้าคุณใช้ provider อื่น, self-hosted model, logging layer, analytics, หรือ human reviewer คุณต้องอธิบายแยกเอง.

อย่าตอบว่า "ไม่ต้องห่วง ปลอดภัย" แบบกว้างๆ ให้ตอบเป็นระบบ: ใช้ provider ไหน, เก็บ log อะไร, ใครเข้าถึง, ลบเมื่อไหร่, และใช้ข้อมูลเพื่อ train หรือไม่.

6. แบบฝึกหัด: Data trust plan

กรอกแผนจัดการข้อมูล แล้วกดประเมิน. ตัวประเมินนี้เป็น JavaScript ในไฟล์ ไม่เรียก AI และไม่ส่งข้อมูลออกไปไหน.

รอแผนของคุณอยู่.

7. Risk level quick check

ประเมินความเสี่ยงข้อมูลแบบเร็ว เพื่อรู้ว่าควรปรึกษาผู้เชี่ยวชาญหรือเพิ่ม guardrail.

ใส่ตัวเลขแล้วกดตรวจ.
